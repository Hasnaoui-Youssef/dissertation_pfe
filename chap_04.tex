\subsubsection{Clock Source}
CoreMark needs the user to implement a timing function in order to provide accurate measures of the CPU's performance.\par
\noindent In embedded systems, Timers are the most efficient way of time measurement. A timer relies on a tick register that decrements periodically, and possibly resets to its maximum value when it reaches 0.\par
\noindent ARM provides a 24-bit system timer SysTick integrated in Cortex-M processor cores.
Manipulating this timer is done via 4 registers :
\begin{itemize}
  \item \textbf{SYST\_CSR} The SysTick Control and Status register enables the SysTick features, such as specifying whether the clock source is external or internal, enabling exceptions and enabling the timer
  \item \textbf{SYST\_RVR} The SysTick Reload Value register specifies the start value of the timer, this is the value to be loaded into SYST\_CVR every time its value reaches 0.
  \item \textbf{SYST\_CVR} The SysTick Current Value register contains the current value of the SysTick counter.
  \item \textbf{SYST\_CALIB} The Systick Calibration Value register indicates the SysTick calibration properties, it is optional and not required to be implemented.
\end{itemize}

\label{sec:systick_interrupt}
When working across the entire STM32 MCU catalog, some families can have a default base clock speed up to 128MHz, therefore, we will be using the maximum value allowed for the counter(\(2^{24}\) Ticks) to reduce the number of interrupts fired during the execution of the benchmark.\par
\noindent Enabling the SysTick timer is done by setting the first bit(index 0) of the SYST\_CSR register to 1.\par
\noindent To keep track of how many times the SysTick timer underflows, we utilize the 32-bit unsigned integer variable provided by ST's HAL library uwTick, we implement the SysTick interrupt handler to increment its value by 1 everytime it fires.\par
\noindent With this method, we now can get the total number of ticks elapsed during the execution of our program with this formula :\par
\(TotalTicks = uwTick * SysTick Reload Value + SysTick Current Value\)\par
\noindent Subsequently, the total time elapsed in seconds would equate to:\par
\textit{Total Time in Seconds} \(= TotalTicks / Processor Frequency\)\par \noindent

\subsection{User Code}
The user code component is made of two pairs of source/header files main and print\_config.
\begin{itemize}
    \item \textbf{Main:} The main header file imports the family's HAL header file. The main file contains the user application's entry point as well as the redefintion of the SysTick\_Handler function for reasons mentioned in the Subsection~\ref{sec:systick_interrupt}, the entry point executes the following sequence then enters an empty infinite loop:
        \begin{itemize}
            \item Calls the HAL initializer function
            \item Calls the print configuration function
            \item Calls the CoreMark benchmark's entry function
        \end{itemize}
    \item \textbf{Print Configuration:} The header file declares the exported functions to be called, the source file provides the implementation, these functions are the UART peripheral initialization and the implementation of the send\_char function show in Figure~\ref{fig:send_char}.
\end{itemize}
\subsection{Modifiable Files}
The modifiable group is a set of files meant to be altered by the user in case they want to change the behaviour and state of the system when executing CoreMark.
\begin{itemize}
    \item \textbf{Linker Scripts:} The linker script defines the memory layout of the application, in the CoreMark application, the default linker scripts are a static sample that offers the bare minimum to get the project running:
        \begin{itemize}
            \item \textbf{Entry Point:} The execution's entry point is different from the user application's entry point, as there's initial setup to be done before entering the user application, this symbol is provided to the linker script in order to know where to start executing.
            \item \textbf{Memory Regions:} The linker script defines two main regions, \textbf{ROM} and \textbf{RAM}, the ROM region, is a 64KB region starting at address 0x08000000, it is meant to hold all text symbols, where as RAM is a 12KB region starting at address 0x20000000, it contains all the data symbols and non initialized variables.
            \item \textbf{Sections:} For the IAR and ARM compilers, regions are equivalent to sections, you only have to define the type of data provided to each region based on permissions (\textbf{R}ead, \textbf{W}rite, E\textbf{x}ecute), however, for gcc, sections are the containers of symbols, we then specificy in which memory region each section resides. Therefore we define the following sections:
            \begin{itemize}
                \item \textbf{Text:} This section contains the instructions of the program. It is typically placed in the ROM memory region  and is marked with execute permissions.
                \item \textbf{Data:} This section stores initialized data, such as global or static variables with predefined values. It is placed in the RAM memory region and requires read and write permissions to allow the program to access and modify these variables during execution.
                \item \textbf{BSS:}  The BSS section contains uninitialized data, similar to the data section, the difference is that the startup file zeroes out this section by default.
            \end{itemize}
        \end{itemize}
    \item \textbf{Startup Files:} The startup file defines the interrupt vector table, defining a default weak implementation for each interruption, also, it defines the entry point function, typically named \textbf{Reset\_Handler}, this function starts by copying the appropriate data from ROM to RAM, and zeroes out the BSS section, afterwards it calls the System Initialization function (System\_Init), libc's initialization function and finally jumps into the user application's code.
    \item \textbf{System Files:} The system header/source files pair implements core system-level initialization functions.
            \begin{itemize}
                \item Configure the clock tree and default peripheral states at reset.
                \item Provide global variables like \texttt{SystemCoreClock} used by CMSIS and HAL drivers.
                \item Setup memory banks and the Vector table location and offset addresses.
            \end{itemize}
\end{itemize}

\subsection{Device Configuration}
The device configuration module only requires the device name as input to construct the necessary information modeling the device, optionally if the strict option is provided, the device name validation happens in this component. \par \noindent
The first step is to load the device.json file. We then construct the regex allowing us to validate the device name, we join the list of keys from the device.json file's objects with the "|" symbol as a separator, the generated regex would look like\newline\texttt{STM32(Family names)[A-Z0-9]\{2\}(Pin Counts)(FLASH Sizes)(Package Type)X[A-Z]?} \newline If valid, exposes the information via a global data object.
\subsection{UART Configuration}
The UART configuration module simply checks each UART flag against a basic regex to conform to the ST drivers' naming convention and fills the data class structure with the information.
\subsection{Extra Configuration}
The Extra configuration modules checks the rest of the flags, ensuring locations exist and the compatibility of flags with the device, (e.g. you cannot enable the processor's cache with a device that doesn't support it), and the generates the last data object to be used in the execution phase.
\subsection{Firmware Management}
This part of the application checks for the validity of the firmware path within the extra configuration's amalgamated path, if it is invalid it prompts the user to download a valid version from the internet, specifically, ST distrubtes the firmware publicly on a GitHub repository following a standard naming convention that can be derived from the device family.
\subsection{File Extraction}
While abstractions speed up the development process and provide a reliable way to communicate with the targeted peripheral, they come at the cost of having a very high memory footprint, this side-effect might cause performance degradations as it effects memory alignments, and might even eat up entire high speed regions such as TCM and cause more cache misses. Therefore, reducing this effect to the maxmium is required, and that is done in this part of the process, since we know which modules we need to add, these are the only ones whose files will be copied.
\par \noindent
For HAL and LL files, they follow a simple naming convention that can be expressed by the following regex : \texttt{stm32(device family)\_(hal|ll)\_(peripheral name)(\_ex)?.c/h}
\par \noindent
For CMSIS files, which are the startup and system source files, as well as the device and system headers, the naming convention relies on the device number, and sometimes on the device metadata as well. The common symbol to use is expressed as \texttt{stm32(device family)(device number)xx}, however as mentioned, the last two characters while most commonly are "x"'s can be the device's exact character, this is because sometimes the startup file or the system file might change behaviour depending on the specificities, and for devices that support TrustZone, each file is distributed in three distinct versions: secure, non-secure and a TrustZone disabled version.
\subsection{File Processing}
Even though most of the driver files can be copied directly into our CoreMark projects, some might not work out of the box, this is due to the nature of the file and what they are meant to serve within STM32 MCU ecosystem.
\begin{itemize}
    \item \textbf{Configuration Header:} The configuration header (stm32\_hal\_conf.h) file is the equivalent to a module manager within the STM32 ecosystem, it contains the macro defines that allows us to enable the needed modules, although they can be provided in the csolution project files, due to their amount, they would end up clustering files with unecessary information, therefore we opted to directly modify the template provided by ST in the STM32CubeFirmware package. This is done by reading the file into memory, reading line by line looking for matches to the modules we want to enable and uncommenting that define (by default all module defines are commented out).
    \item \textbf{ARM Compiler Startup File:} For HAL where startup files are not unified C language files but are instead toolchain specific assembly files, the ARM Compiler toolchain has the quirk of defining the heap and stack sizes in the startup file instead of the linker file, therefore we have to manually change the symbols to values that allows CoreMark to run without memory overflows.
    \item \textbf{Linker Files:} Although we use generic linker scripts for HAL projects, HAL2 based projects rely on the linker scripts provided within the STM32CubeFirmwareV2 package for the device, therefore, we manually change the stack and heap sizes defined within these files.
\end{itemize}
\subsection{Template Generation}
The last part of the execution phase is to generate files based on the template resources we have, the standard library for Golang provides a very handful package "text/template" that we will be using, working with this package is very straight forward, it exposes an interface to load a template file, and then executing the template consists of making use of an other interface that takes the template handle and a data object as arguments, we then save the result to the desired file.
\subsection{Build Information Generation}
After generating the project, the last step before being able to build it is to generate build information, this is in the form of the build information files of the CMSIS-Toolbox such as cbuild.yml and cbuild-run.yml, as well as the RTE and .cmsis folders containing device specific information for the build stage. This is done by invoking the CMSIS-Toolbox utility \textbf{cbuild} with the setup subcommand and the csolution file as an argument.
\subsection{Error Handling And Resilience}
With the number of variables, options, external dependencies and interactions with the file system, multiple parts of the project can fail, therefore maintaining a robust error handling system is crucial in order to handle all cases.
\begin{itemize}
    \item \textbf{Invalid Flags:} for the crucial flags being the device name and UART information, errors will result in a non-working project, therefore the absence of these flags or their invalidity means that the program will terminate without proceeding any further, for all other flags, we simply revert to the default values and continue operating.
    \item \textbf{Missing Resources:} As they are an integral part of the program, not being able to load them means the application will not be able to operate.
    \item \textbf{Invalid Firmware:} In the case that the firmware is not available, or if for some reasons, doesn't provide the appropriate files, the program will fail gracefully, reverting all the changes it has done to the project directory.
\end{itemize}
\subsection{Extensibility And Future Proofing}
\label{sec:fut_ext}
cm\_gen was architectured in a way that allows for extensions, as it is meant to work on any STM32 device including those that are still in development and aren't even available as chips, therefore being future-proof is a must in these circumstances. This is done in two ways throughout the application:
\begin{itemize}
    \item  \textbf{External Resources:} By providing device and library metadata in external files, we allow the user to update and customize them without having to modify the application's source code.
    \item  \textbf{Interfaces:} Providing implementations on how to extract files and process them depends a lot on the context, by providing a generic interface that the mediator within the application invokes in an abstracted way allows us to extend this interface in the future with different implementations that adhere to new needs that aren't present at the time.
    \item  \textbf{Dynamic templating:} The data object passed to templates is a comprehensive union of all metadata within the data objects of the application as well as extra information derived during the earlier steps of the execution phase, therefore, the tempalte files can be modified in any way the user sees fit as long as it adheres to the dynamic variables' naming convention and making sure that these variables are provided by the data object.
\end{itemize}
\section*{Conclusion}
In this chapter, we detailed the comprehensive process of implementing and automating the CoreMark benchmark for STM32 devices. The project began with porting CoreMark to STM32 microcontrollers, focusing on configuring the SysTick timer for precise performance measurements and implementing the \texttt{ee\_printf} function for result reporting via UART. We then transitioned to integrating the csolution project structure, which streamlined project setup by defining source files, include paths, and linker scripts. The development of the \texttt{cm\_gen} tool marked a significant milestone, enabling fully automated generation of CoreMark projects tailored to specific STM32 devices. This tool leveraged device metadata, UART configurations, and template-based file generation to reduce manual effort and minimize errors. Validation across all STM32 MCU families confirmed the robustness of the solution, with results benchmarked against manual setups. The modular architecture of \texttt{cm\_gen}, combined with its extensible design, ensures compatibility with future STM32 devices and evolving requirements. This work establishes a scalable, efficient framework for performance benchmarking, significantly enhancing development workflows for ST's embedded systems.
